# Fix cash balance: use Evotor /state endpoint instead of manual calc

> Generated by Deep Copilot · Plan mode
> 2026-07-02T09:33:46.511Z

## 目标 / Goal
Заменить ручной расчёт остатка наличных (getCash/getCashByShops) на прямой запрос официального эндпоинта Evotor /state, который возвращает точный остаток кассы без арифметических ошибок.

## 方案 / Approach
Текущий getCash пытается вычислить остаток вручную через Z_REPORT + SELL/PAYBACK/CASH_INCOME/CASH_OUTCOME за 7 дней. Это хрупко и неверно, если Z_REPORT старше 7 дней. Правильный способ — Evotor API: GET /api/v1/inventories/stores/{storeUuid}/state → cashRegisterState.cash (точный остаток наличных напрямую из кассы).

## 步骤 / Steps
1. **Добавить метод getStoreState в класс Evotor** — GET /api/v1/inventories/stores/{storeUuid}/state. Кэширование не нужно — каждый вызов должен возвращать актуальный остаток.
2. **Переписать getCash на использование getStoreState** — Вместо 50 строк ручного расчёта — 1 запрос к /state, возврат cashRegisterState.cash.
3. **Переписать getCashByShops аналогично** — Параллельные запросы getStoreState по всем магазинам через Promise.all.
4. **Конвертация копеек в рубли** — API возвращает копейки. Убедиться, что фронтенд ожидает рубли, и добавить деление на 100.
5. **Проверить компиляцию и тесты** — npx tsc --noEmit, npx vitest run

## 涉及文件 / Affected files
- `packages/backend/src/evotor/index.ts` — Добавить getStoreState, переписать getCash и getCashByShops

## 风险与缓解 / Risks
- API /state может работать не для всех версий тарифов Evotor — нужен fallback на старый метод
- Значения из /state могут быть в копейках — нужна конвертация в рубли для совместимости с фронтендом
- Эндпоинт не кэшируется — каждый вызов идёт в Evotor API (это плюс для точности, но может замедлить загрузку)
- Фронтенд ожидает Record<string, number> с названиями магазинов — сигнатура не меняется
