## 目标 / Goal
Перенести всю логику синхронизации из отдельного пакета @evo-app/sync в backend Worker, чтобы sync и backend использовали одну базу D1 (d1-bot) и жили в одном Cloudflare Worker — HTTP-эндпоинты + cron-триггеры.

## 方案 / Approach
Backend Worker уже содержит класс Evotor и многие утилиты. Нужно добавить в backend: (1) cron-триггеры в wrangler.toml, (2) недостающие DB-функции из sync/util.ts в backend/src/utils.ts, (3) cron-задачи из sync/index.ts в backend/src/index.ts, (4) недостающие таблицы D1. Затем удалить пакет @evo-app/sync.

## 步骤 / Steps
1. **Добавить cron-триггеры в backend wrangler.toml** — Добавить [triggers] с crons = ['30 3 * * *'] для ежедневного запуска в 3:30
2. **Перенести уникальные DB-функции sync → backend utils.ts** — Функции: createProductsTableIfNotExists, updateOrInsertData, getProducts, updatePlanDb, getPlanData, createSalaryTable, saveSalaryData, initializeSaleByPlanTable, upsertSaleByPlanData, getSalesDataByDate, updateOrInsertDataS, delay, dataGroup. Проверить конфликты с существующими getPlan/updatePlan.
3. **Перенести cron-задачи в backend src/sync/cron.ts** — Функции: updateProducts, updateProductsShope, getDocuments, getDataForCurrentDate, updatePlan_, updateDataSaleByPlan. Вынести хардкод (Vape group UUIDs) в константы.
4. **Добавить scheduled() handler в backend index.ts** — Импортировать cron-задачи, добавить export default { fetch: app.fetch, scheduled } в index.ts
5. **Сверить бизнес-логику Evotor (adjustSales: 3% vs 5%)** — В sync adjustSales множитель 1.03, в backend 1.05. Нужно решить, какое значение верное, и унифицировать.
6. **Обновить D1-схему — добавить недостающие таблицы** — Таблицы из sync, которых нет в backend: products, shopProduct, salaryData, SaleByPlan. Создать миграции Drizzle или SQL-файлы.
7. **Проверить TypeScript-компиляцию backend** — Запустить tsc --noEmit для backend, убедиться что нет ошибок.
8. **Удалить пакет @evo-app/sync из workspace** — Удалить packages/sync, убрать sync:dev/sync:deploy скрипты из корневого package.json, запустить pnpm install.

## 风险与缓解 / Risks
- Дублирование функций: getPlan и updatePlan есть в обоих util-файлах с разной логикой. Нужно аккуратно слить, сохранив обе версии под разными именами или унифицировав.
- Разная бизнес-логика adjustSales: 3% в sync vs 5% в backend. Нужно уточнить, какая версия правильная.
- Разные сигнатуры D1-хелперов: sync использует db напрямую, backend — через c.get('db') / c.env.DB. Нужно унифицировать.
- В sync есть хардкод dataGroup (большой массив из 200+ продуктов). Его нужно либо перенести, либо понять, используется ли он ещё.
- После удаления sync package может сломаться deploy-пайплайн, если он был завязан на отдельный Worker products-work.
