# Telegram-хранилище фото открытия вместо R2

> Generated by Deep Copilot · Plan mode

## 目标 / Goal
Заменить R2-хранилище фотографий открытия на Telegram: бэкенд отправляет фото боту через sendPhoto, сохраняет только file_id в D1, а при запросе — дёргает getFile и отдаёт URL. Фото живут на серверах Telegram.

## 方案 / Approach
Два метода на бэкенде: (1) sendPhotoToTelegram — multipart-запрос к Bot API, возвращает file_id. (2) getPhotoUrl — по file_id получает URL через getFile. Слой хранения — D1-таблица opening_photos, в которой только метаданные + file_id. Фронтенд не меняется — он уже шлёт POST /api/uploads/upload-photos, просто endpoint перестаёт быть заглушкой.

## 步骤 / Steps
1. **Создать sendPhotoToTelegram утилиту** — packages/backend/utils/sendPhotoToTelegram.ts: принимает Buffer, filename, chatId, BOT_TOKEN → fetch sendPhoto (multipart) → возвращает file_id
2. **Добавить TELEGRAM_STORAGE_CHAT_ID в env** — packages/backend/wrangler.jsonc + types.ts: ID приватного канала, куда бот шлёт фото (не виден пользователям)
3. **Создать D1 таблицу opening_photos** — Миграция или createTable: id, shop_uuid, user_id, date, category, slot_index, telegram_file_id, created_at. Уникальный ключ (shop_uuid, date, category, slot_index)
4. **Реализовать POST /api/uploads/upload-photos** — Заменить заглушку: принять multipart (file, category, userId, shopUuid, fileKey), сжать если нужно, отправить в Telegram, сохранить file_id в D1, вернуть { success: true }
5. **Создать GET /api/openings/photos** — Принимает ?date=X&shopUuid=Y, читает file_id из D1, для каждого вызывает getFile → URL, возвращает массив { category, slot, url }
6. Верификация — curl оба метода

## 风险与缓解 / Risks
- Фото до 20 MB через sendPhoto (Telegram лимит) — наши сжатые JPEG 1200×800 укладываются
- file_id постоянный только в рамках одного бота — при смене токена старые file_id станут невалидны
- Приватный канал должен существовать и бот должен быть в нём админом — иначе sendPhoto вернёт 403
